# Gemma用 採点基準 作成レポート

## 概要
全21課題について、ローカルVLM（Gemma）が**機械的に採点できる**採点基準（`grading_criteria/*.md`）を作成・整備した。全サンプル提出物（各8〜11人分、計約200ファイル）を画像化して実際に採点し、その結果をもとに「主観判断ではなく、画像から観察できる事実にYes/Noで答え、その結果から点数を機械算出する」設計に統一した（既存の事前課題1の方針を全課題へ展開）。

## 設計の基本方針
- VLMに「良い／悪い」を主観で判断させず、**観察項目（Yes/No）**を答えさせる
- 観察結果から**決定木で機械的に点数を算出**（プロンプト内で計算手順を明示）
- `grader.py` が要求するJSONキー（`点数` / `観点別評価` / `評価理由` / `フィードバック`）に厳密準拠
- ファイル名は `assignment_config.json` のキーと完全一致（最終版は全角スペース）
- 配点は calibration の満点に厳密準拠：**事前課題=3点／最終版=2点／リフレクション=9点（3観点×3点）**

## 課題タイプ別の採点ロジック
- **事前課題2〜7（3点満点）**: 課題固有の必須観察項目（変数/条件/トレース表/デバッグ観点 等）＋量＋質を観察。「量不足 or 必須項目が半分以下 → 1点」「必須ほぼ充足＋その課題の★要素 → 3点」「中間 → 2点」。★要素は課題ごとに設定（例: 課題3=境界値120とトレース、課題4=投入金額/在庫の変化トレース表、課題6=誤り箇所3つ以上＋理由、課題7=エレベーター＆レジの2題材）。
- **最終版1〜7（2点満点）**: 赤ペン加筆／青ペン加筆／加筆の質の3観察。赤青両方＋意味ある加筆=2点、片方のみ=1点、無し=0点。
- **リフレクション1〜7（9点満点）**: 気づき／差分／再構成の3観点を各1〜3点で観察項目から算出し合計（3〜9点）。全週で同一ルーブリック（テーマ語のみ差し替え）。

## 実サンプル採点の結果（校正データ・要約）
- **事前課題1〜7**: いずれも提出層のレベルが高く、多くが3点（6観点・トレース表・考察まで充実）。1〜2点は量不足・動作と状態の混同・題材1つのみ等の少数。
- **最終版**: 赤の加筆はほぼ全員に明瞭。青のみ無い提出が散見され、その場合1点。
- **リフレクション**: 3〜9点に分布。差分（他者比較）の具体性と再構成（今後の具体的活用）の有無が点差の主因。

## Gemmaで採点する際の重要な注意点（要対応）
1. **用紙の自動回転**: 事前課題（横向き）でも縦向き（横倒し）でスキャンされた提出が混在。`grader.py` の `assignment_config.json` 向き補正で概ね対応できるが、向きがおかしいと観点ラベルを読めない。
2. **解像度・コントラスト**: 薄い鉛筆書きや暗い写真は150dpiで判読困難。**250〜300dpi＋必要ならコントラスト補正**を推奨。逆に巨大画像（4000px超）はAPIに拒否されることがあり、`grader.py` の縮小（max_size=1568）で吸収。
3. **青と黒の誤認（最終版で最大のリスク）**: 淡い青の見落とし・黒を青と誤認が起きうる。基準mdに色判別の注意を明記済み。緑ペン・蛍光マーカーの併用者がいるため、赤/青判定に混ぜないこと。
4. **重複・別様式の提出**: 同一内容の再スキャンや、所定様式でない変則提出が一部存在。学生単位での重複排除・様式チェックを前段に入れると安定。
5. **散文 vs ラベル付き**: 観点をラベル無しの散文で書く提出は人間でも判定が割れる。基準は「ラベルが無くても内容から読み取れればYes」と明記済み。

## 検証結果
- 全21ファイルが存在し、`assignment_config.json` の全キーと**完全一致**。
- 全ファイルに必須JSONキー（点数・観点別評価・評価理由・フィードバック）と出力テンプレートを確認。
- 配点レンジが calibration の満点と一致（3／2／9）。
- 採点ロジックをコード化して代表ケースで検証し、サンプル採点の点数を再現・全レンジを網羅することを確認。
